## 三、记忆更可控

### 3.1 记忆面板（看 / 改 / 删画像与锚点）

#### 产品目标

用户在「全自动写入」前提下仍感到**被尊重、可纠错**：能看见系统记住了什么，能改正错误或删除不想被沿用的内容。

这不是「每次写入弹确认」，而是 **事后可纠正**（业界 Ash / Replika Memory tab / Rosebud 共识）。

#### 用户故事

1. 作为用户，我打开「伴侣记得什么」，能看到画像摘要、锚点列表、（可选）触发/资源摘要。  
2. 作为用户，我发现画像写错了关系状态，能编辑该段并保存，之后对话不再沿用错误事实。  
3. 作为用户，我不想再被提起某次事件，能删除或封存对应锚点。  
4. 作为用户，我误删后可以接受「不可恢复」或「软删除 7 天」（实现时二选一，建议软删除）。

#### 信息架构（建议）

- 入口：账号菜单「我们如何记得你」升级为二级页，或「我的 → 记忆」。  
- 区块建议：  
  - **关于你（画像）**：按结构化字段展示（见 3.4），也可先展示现有整段 Markdown。  
  - **重要事件（锚点）**：卡片列表：事件名 / 最初看法 / 当前看法 / 更新时间；操作：编辑、删除、勿提。  
  - **说明**：再次强调「后台自动更新；你可随时纠正；不是诊疗」。  

本阶段**不要求**做完整「事实记忆 Mem0 层」UI。

#### 数据与 API（建议）

| 能力 | 建议 API | 备注 |
|------|----------|------|
| 读画像 | `GET /api/memory/profile` | 现有 `profiles` 表 |
| 改画像 | `PUT /api/memory/profile` | 校验长度；写审计日志（谁改了什么） |
| 列锚点 | `GET /api/memory/anchors` | 分页 |
| 改锚点 | `PATCH /api/memory/anchors/:id` | 字段：event_name, initial/current_thought |
| 删锚点 | `DELETE /api/memory/anchors/:id` | 软删优先 |
| 勿提标记 | `PATCH ... { muted_until \| sealed: true }` | 见 3.5 |

前端：新页或 Drawer；编辑用大文本区；删除二次确认。

#### 与夜间任务的冲突策略

- 用户手动编辑后，该字段/锚点标记 `user_locked_at` 或 `manual_override=true`。  
- 夜间任务：**跳过锁定字段**，或只追加未锁定块，避免「用户改完夜里又被盖回去」。  
- 产品文案：说明「你改过的部分，我们会优先尊重你的版本」。

#### 验收标准

- [ ] 登录用户可查看本人画像与锚点，看不到他人数据。  
- [ ] 编辑画像后，下一轮 `buildChatMessages` 注入的是新内容。  
- [ ] 删除锚点后，`match_anchors` 不再返回该条。  
- [ ] 空画像有友好空态，不展示原始「尚未生成」恐吓文案。  

#### 明确不做（本项内）

- 逐条对话「加入记忆」确认流。  
- 社交分享记忆。  
- 管理员代用户改记忆（除非运维工具另案）。

### 3.2 周摘要（weekly summary）

#### 产品目标

让系统能回答「这几周反复出现的关系位置是什么」，支撑后续「模式回顾」，而不只依赖近 30 条原文 + 日摘要 top2。

#### 现状缺口

- 已有 `summaries` 的 `daily` 类型与 `match_summaries_daily`。  
- 周/月摘要表结构或任务未作为生产主路径跑通（以代码当时状态为准，实施前再核对）。

#### 生成规则（建议）

| 项 | 建议 |
|----|------|
| 触发 | 夜间任务；每周一凌晨汇总「上一个自然周」 |
| 输入 | 该周 daily summaries + 抽样对话 + 该周日记标题/摘要 |
| 输出 | 800–1500 字内：情绪起伏、重复主题、关系场域、需跟进的一句 |
| Embedding | 写入后向量化，供 chat 检索 |
| 去重 | 同一 `user_id + week_start` 唯一 |

#### Chat 注入（建议）

- 检索：`match_summaries` 扩展为按 type 配额，例如 daily×1 + weekly×1；或独立 `match_summaries_weekly`。  
- Prompt 块：`[历史摘要]` 内标注日期尺度，如 `周摘要(2026-07-07~07-13): …`。

#### 验收标准

- [ ] 连续使用 ≥7 天的测试账号能生成至少 1 条周摘要。  
- [ ] 对话检索能命中周摘要并出现在 prompt。  
- [ ] 无对话无日记的空周不生成垃圾摘要。

#### 月/年摘要

- 本文仅定周摘要；月/年可在周摘要稳定后按同样模式扩展，不提前做。

### 3.3 检索加权（语义 + 近因 + 类型配额）

#### 产品目标

减少「捞到很旧、不相关、或某一类记忆占满上下文」的问题，提高「被准确看见」感。

#### 现状

- 用户消息 embedding → `match_summaries_daily` top2、`match_anchors` top1、`match_diaries` top2。  
- 纯向量距离，无时间衰减，无类型配额（日记接入后类型变多，更需要配额）。

#### 建议算法（可迭代）

对候选集（摘要 / 锚点 / 日记）分别取较多候选（如各 5 条），再打分重排：

```text
score = w_sem * semantic_similarity
      + w_recency * recency_score(updated_at)   // 如指数衰减，半衰期 30～90 天可配
      + w_type_bonus                           // 可选：用户本轮明显在谈「旧事件」时提高锚点权重
```

**类型配额（硬约束，建议默认）**

| 类型 | 注入上限 | 说明 |
|------|----------|------|
| 近期对话 | 保持现有条数或略减 | 过长时可先摘要化（另项） |
| 日摘要 | 1 | |
| 周摘要 | 0～1 | 有则优先于第二篇日摘要 |
| 锚点 | 1 | |
| 日记 | 1～2 | 与当前已实现的日记 RAG 对齐 |
| 画像 | 全文或结构化块 | 始终注入，但宜控制长度 |

#### 配置

环境变量示例（实施时再落代码）：

```env
CHAT_RAG_DAILY_LIMIT=1
CHAT_RAG_WEEKLY_LIMIT=1
CHAT_RAG_ANCHOR_LIMIT=1
CHAT_RAG_DIARY_LIMIT=2
CHAT_RAG_RECENCY_HALF_LIFE_DAYS=60
```

#### 验收标准

- [ ] 构造「旧日记很相似、新日记略相似」的夹具数据，重排后更倾向合理近因（记录 A/B 结果）。  
- [ ] Prompt 中各类块不超过配额。  
- [ ] 无 embedding 的日记/摘要被安全跳过（已有 `embedding is not null` 条件则保持）。

#### 明确不做（本项内）

- 上完整 Graphiti/Zep 图谱。  
- 跨用户检索。

### 3.4 画像结构化与增量更新

#### 产品目标

降低「整篇重写画像」导致的漂移、难纠错、token 浪费；让记忆面板可按块编辑。

#### 建议结构（Markdown 块或 JSON，二选一；推荐结构化 JSON + 渲染 Markdown）

对齐舆情/产品文档中的维度，建议字段：

| 块 ID | 名称 | 内容要点 |
|-------|------|----------|
| `attachment` | 依附与自我 | 自我价值、羞耻敏感、4F 倾向（描述性，非诊断） |
| `relationships` | 关系场域 | 原生家庭 / 伴侣 / 职场 / 友谊 分节 |
| `triggers` | 触发点 | 情境、身体信号、典型想法（可与 4.1 联动） |
| `resources` | 资源库 | 有效安抚、可信关系、边界经验 |
| `timeline` | 重大经历 | 简表，忌流水账 |
| `notes` | 其他 | 用户明确说过的偏好（称呼、勿提主题指针） |

#### 夜间更新策略

1. **增量补丁**：模型输出 JSON Patch 或「仅修改的块」，而不是整篇覆盖。  
2. **慢更新**：高频变动块（如 triggers）可每周；稳定块（attachment）可 1–2 周。  
3. **证据引用**：每条更新附 `source_refs`（message_id / diary_id / date），便于审计与面板「为何记得」。  
4. **冲突**：新信息与旧块矛盾时，写入「当前看法」并保留「曾相信」，呼应锚点 initial/current。  
5. **尊重锁定**：见 3.1。

#### 迁移

- 现有 `profiles.content` 纯文本：首次结构化时跑一次性 migration prompt，失败则保留原文块 `legacy`。  

#### 验收标准

- [ ] 连续两晚任务后，未锁定块可更新且全文长度受控（设 max chars）。  
- [ ] 用户锁定块不被覆盖。  
- [ ] Chat 注入可读（块标题清晰）。

### 3.5 「勿提 / 封存」

#### 产品目标

创伤话题允许**选择性不进入主动提及**：系统仍可「知道」（或按产品选择真正删除），但对话中默认不主动提起。

#### 两种强度（实施时需再确认一种默认）

| 模式 | 行为 | 适用 |
|------|------|------|
| **勿提（mute）** | 仍存库；RAG 降权或过滤；prompt 增加「勿主动提及：…」 | 暂时不想聊 |
| **封存（seal）** | 不对 chat RAG 可见；夜间任务不更新该条；仅记忆面板「已封存」可见 | 强边界 |

建议默认：**锚点与画像块支持勿提；封存作为二次确认的升级操作**。

#### 数据模型（建议）

```text
anchors.muted_until timestamptz null
anchors.sealed_at timestamptz null
profile_blocks[].muted / sealed
```

Chat 组装：

- `match_*` SQL 增加 `sealed_at is null` 与（可选）mute 过滤。  
- System/user 可注入一行：`用户希望暂时不要主动提起：……`（仅 mute 列表，避免把封存内容又写进 prompt）。

#### 验收标准

- [ ] 勿提后，多轮对话不主动点名该事件（人工评测 + 关键词抽检）。  
- [ ] 用户在面板取消勿提后恢复可检索。  
- [ ] 封存条目不出现在 `[重要锚点]` / `[相关日记]`（若日记也支持封存）。

#### 伦理与文案

- 说清楚：勿提 ≠ 失忆；封存更接近「对话里当不存在」。  
- 不做「永久对模型隐藏但运营可见」的双轨，除非合规需要另案。

## 四、产品差异化能力（非立刻必做）

### 4.1 触发地图 / 资源库

#### 产品目标

把舆情报告中的 **Trigger Map / Resource Bank** 从「埋在画像长文里」变成**用户可见、可改**的一等公民，并在闪回边缘时（危机正式做之前）仍可用于温和提醒「你以前什么对你有用」。

#### 功能说明

- **触发地图**：列表项 = 触发情境 + 身体/情绪信号 + 常见想法 +（可选）关联锚点。  
- **资源库**：列表项 = 有效策略/人/物/句子；来源标记「你写下的」或「我们一起发现的」。  
- 夜间任务可**提议新增**，用户可在面板确认编辑（与「全自动写入」兼容方式：自动写入草稿区，或直接写入但可删）。

#### UI

- 记忆面板子 Tab，或独立「地图」弱入口（避免首页仪表盘化）。  
- 卡片操作为主；不做复杂可视化强制。

#### 验收（未来）

- [ ] CRUD 完整；chat 可按配额注入 0～1 条相关 trigger/resource。  
- [ ] 用户删除后不再注入。

---

### 4.2 模式回顾（周）

#### 产品目标

主动但低频地提供「见证」：例如每周一次轻量卡片——「这周你多次谈到被比较/被选择」，点击可「跟伴侣聊聊这个」→ 填入输入框（不强制发送）。

#### 依赖

- 周摘要（3.2）稳定。  
- 单会话：回顾是**卡片/Inbox**，不是新会话。

#### 频率与克制

- 默认每周最多 1 条；可关闭。  
- 文案避免成绩单与羞耻（禁止 streak 羞辱）。

#### 验收（未来）

- [ ] 有周摘要时生成回顾；用户可忽略、关闭、或一键带入对话。  
- [ ] 关闭后不再打扰。

---

### 4.3 可选练习卡片

#### 产品目标

在深度对话之外提供**可选**的 grounding / 边界话术等微工具，降低「只会聊不会落地」的缺口，同时不把产品做成 Wysa 式流程强迫。

#### 形态

- 对话中由模型或规则建议「一张卡片」（呼吸、感官落地、暂停关系对话的一句话）。  
- 用户可展开练习 / 忽略；**绝不**阻断输入。  
- 练习库内容需 trauma-informed 审阅，避免暴露作业。

#### 明确不做

- 强制每日练习、游戏化积分。  
- 危机干预替代品（危机仍单独立项）。

## 七、验收与实验原则（通用）

1. **先信任后花哨**：记忆面板与勿提优先于练习卡片。  
2. **单会话不破坏**：任何「回顾/练习」都是叠加层，不引入会话列表。  
3. **自动写入 + 可纠正**：不回退成确认写入，除非产品重新决策。  
4. **评测**：改检索/画像后，用固定长自述 + 含日记夹具账号做前后对比（字数不是唯一指标；看是否误提封存内容、是否引用错事实）。  
5. **文档同步**：开工某项时，在该项章节顶部把状态改为「进行中 / 已上线」，并链到 PR。
